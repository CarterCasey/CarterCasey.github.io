Scalability, Performance, and Caching - Slides

Terminology

  • Performance - speed and efficiency.
  • Scalability - "low barriers to growth," can get big without problems.
  • Availability - there when you need it.

  • Reliability - never doing anything wrong.

  • Throughput - rate at which whole system does work.

  • Response Time - time after a request that you get a response.
  • Latency - time spent doing nothing.

Tension between throughput and response time - between system and single user.

Know How Fast Things Are

(See slides)

Hard Disk Slowness

  • A lot of the latency in a system is in hard disks.
  • We have to move a needle out over part of a disk, put it down, then pick it up again.
  • It's fast, but mechanically fast - which means it's hugely slower than cpus.

Paralellism

Amdahl's Law

  • Parallel Processing won't scale up.
  • While we can do a lot of work on the parallelizable phase, but some phases can't be done in parallel.

  • I/O will usually be slow.

  • He was mostly right, but then people started working on much larger problems.

Web Performance and Scaling

  • The Web has to be huge, so scalability is more important than performance (but we shouldn't sacrifice perfomance just for that).

Caches

  • Locality:

    • We want to work with things that are nearby.
    • This is effective because there's a small amount of data we usually care about.

  • Why is caching hard?

    • Things change. You don't want to show old, incorrect information.

  • Cache invalidation on multi-core machines:

    • Caches have to communicate with each other in order to make sure that you never get a bad read.

  • Consistency vs Performance

    • It's really easy to be fast if you don't care about the rest of the system.
