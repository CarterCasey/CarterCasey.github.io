Security - Slides

Thompson Paper

  • Any system you use can theoretically change what you produce with this.
    • This is especially important with compilers, linkers, etc. Any program that handles other programs.

Encryption

  • Symmetric Key: both sides use the same key.
  • Asymmetric Key: each side uses different key, paired in a specific way.
    • The keys can't be switched - the encryption key can't be used to decrypt it.
  • This leads us to public key crypto.

Public Key Cryptography

  • If the encryption key can't be used to decrypt, then we can just make it public.
    • If you want to send me some data, you can use that public key.

  • I, on the other hand, have the private key. It's private, so ideally no one else has it.

    • That means I should be the only person who can decrypt the information.

  • One danger: fraudulent public keys.

Certificates

  • Certificate is a signed public key.
  • There's a structure to these certificates:
    • A signature has to come from somewhere that's trusted.
    • They can't all come from the same single place, so there are root certificate authorities.
    • It feels a lot like DNS to me.

HTTPS

  • Use transport level security (TLS, SSL) to check certificates.
  • We have a problem with the slowness of public key cryptography.
  • To solve this, we do something like sending an encrypted symmetric key, then using that for the rest of the transfer.

Proxies and MitM

  • Web is multi-tiered system that uses proxies.
  • If we have a malicious proxy in the middle, it can change the data you're expecting.

Principles

  • Who are we trusting?
  • What are the dangers of trusting them?
  • How do we know if we trusted the wrong source?
  • Can we revoke the trust?

Real Attacks (that work)

  • Freezing RAM chips can retain data for longer.
  • Micro-timing SSH response to figure out key.
